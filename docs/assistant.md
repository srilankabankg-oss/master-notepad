# Ассистент — AI-агент с RAG

## Назначение

**Ассистент** — интеллектуальный агент, который отвечает на вопросы пользователя в свободной форме, анализируя данные системы Master Notepad. Пользователь спрашивает как человека («Я на объекте X, какие чек-листы нужны?»), ассистент понимает контекст, извлекает релевантные данные через RAG (Retrieval-Augmented Generation) и даёт конкретный ответ на основе фактов из системы.

Это переход от «поиска по меню» к прямому вопросу — главный интерфейс взаимодействия для всех ролей.

## Принцип работы

1. **Пользователь задаёт вопрос** в свободной форме на русском языке
2. **RAG-конвейер** извлекает релевантные фрагменты из базы данных системы (подрядчики, отзывы, события, чек-листы, протоколы, опросы, комментарии)
3. **LLM формирует ответ** на основе извлечённого контекста — только по фактам из системы, без выдумывания
4. **Ответ содержит ссылки** на конкретные записи, из которых взят ответ

## Источники данных (RAG)

Ассистент индексирует все сущности системы:

| Сущность | Таблица | Что извлекается |
|----------|---------|-----------------|
| Подрядчики | `subcontractors` | Название, специализация, контакты |
| Отзывы | `reviews` | Текст + оценка 1-10 |
| Журнал событий | `contractor_events` | Тип (positive/violation/info), описание, дата |
| Комментарии | `comments` | Дополнительная информация |
| Чек-листы | `checklists` | Названия, пункты, тип (org/personal) |
| Предложения | `checklist_suggestions` | Текст предложений и статус |
| Протоколы совещаний | `meeting_protocols` | Повестка, решения, участники |
| Опросы | `surveys` + `survey_responses` | Вопросы и ответы сотрудников |
| Сотрудники | `employees` | ФИО, должности |

## Сценарии использования

### Сценарий 1: Подготовка к приёмке на объекте

> «Я на объекте ЖК Восток, принимаю фундамент. Какие чек-листы есть по фундаменту и что в них важно проверить?»

**Ответ ассистента:**
- Находит чек-листы, содержащие в названии «фундамент» (организационные и личные)
- Показывает пункты этих чек-листов
- Дополнительно: предлагает посмотреть журнал событий подрядчиков, работавших на фундаментах — были ли типовые нарушения, которые стоит проверить
- Если есть предложения по улучшению (status: `approved`) для этих чек-листов — включает их в ответ

### Сценарий 2: Выбор подрядчика на тендер

> «Нужен подрядчик на электрику, кто у нас был за последний год? У кого меньше всего нарушений?»

**Ответ ассистента:**
- Фильтрует подрядчиков по специализации «электрика»
- Смотрит `contractor_events` за последний год: считает нарушения (type: `violation`) по каждому
- Смотрит рейтинг (AVG `reviews.rating`)
- Ранжирует: сначала чистые, потом с нарушениями
- Для каждого показывает ключевые факты из отзывов и событий

### Сценарий 3: Разбор конфликта

> «Подрядчик СтройМонтаж говорит что мы его не предупреждали о сроках. Что у нас есть по нему за май 2025?»

**Ответ ассистента:**
- Фильтрует `contractor_events` по `subcontractor.name = 'СтройМонтаж'` и периоду май 2025
- Показывает хронологию: события, протоколы совещаний (если обсуждались сроки)
- Находит отзывы, где упоминаются сроки (по тексту `reviews.content`)
- Выдаёт конкретные даты и записи: «15 мая — событие violation "сорвал поставку арматуры на 3 дня" (записал Иванов И.И.)»

### Сценарий 4: Типовые ошибки по специализации

> «Какие типовые ошибки проектирования у нас были по автоматизации? Хочу добавить в чек-лист.»

**Ответ ассистента:**
- Ищет `contractor_events` типа `violation` по подрядчикам со специализацией «автоматизация»
- Анализирует текст описаний событий, группирует похожие
- Показывает уже одобренные предложения (`checklist_suggestions` status: `approved`) по релевантным чек-листам
- Предлагает неучтённые нарушения как кандидаты в чек-листы

### Сценарий 5: Контекстная справка по подрядчику

> «Расскажи всё что знаешь про ТехноСтрой»

**Ответ ассистента:**
- Сводка по всем сущностям: карточка подрядчика, рейтинг, количество отзывов, хронология ключевых событий, последние протоколы совещаний, результаты опросов
- Сжатое резюме: «Надёжный подрядчик, 3 нарушения за год, все мелкие. Рейтинг 7.8/10 на основе 5 отзывов. Последнее совещание — 10 апреля.»

## Ценность

| Для кого | Что даёт |
|----------|----------|
| **Руководитель** | Мгновенная сводка вместо ручного сбора информации по разделам. Решение за 30 секунд вместо 10 минут. |
| **Инженер / Прораб** | Прямо на объекте спрашивает чек-листы, не роясь в меню. PWA на телефоне + голосовой ввод. |
| **Делопроизводитель** | Быстро найти протокол или участников совещания, не помня точных дат. |
| **Новый сотрудник** | Вход в контекст: «расскажи про подрядчиков по электрике» — и он уже в курсе опыта коллег. |

## Технический дизайн

### Архитектура микросервиса

Ассистент — отдельный микросервис на **Python/FastAPI** (порт 3002). Вынесен из основного Node.js-бэкенда для изоляции ML-зависимостей и возможности независимого масштабирования.

```
assistant/
├── src/
│   ├── main.py            # FastAPI entry (порт 3002)
│   ├── api/
│   │   ├── ask.py         # POST /api/ai/ask — основной эндпоинт
│   │   └── reindex.py     # POST /api/ai/reindex/{entity} — переиндексация
│   ├── rag/
│   │   ├── embedder.py    # Векторизация (multilingual-e5-base, 768-dim)
│   │   ├── retriever.py   # Гибридный поиск: 0.7 cosine × 0.3 BM25
│   │   └── ingester.py    # Индексация: запрос данных из основного API → чанки → embeddings → pgvector
│   ├── llm/
│   │   └── provider.py    # Адаптер к OpenAI API (заменяем через конфиг)
│   └── db/
│       └── vector_store.py # Клиент pgvector: HNSW-индекс, similarity search
├── Dockerfile
├── requirements.txt
└── pyproject.toml
```

### Стек
| Компонент | Технология |
|-----------|------------|
| Рантайм | Python 3.11+ / FastAPI + uvicorn |
| Векторная БД | pgvector (таблица `embeddings`, HNSW-индекс, cosine distance) |
| Embedding модель | `intfloat/multilingual-e5-base` (768-dim, ONNX для прода) |
| LLM | OpenAI API (`gpt-4o-mini` по умолчанию, заменяем через `LLM_API_URL`) |
| Поиск | Гибридный: vector similarity (cosine, вес 0.7) + BM25 keyword (вес 0.3) |
| Развёртывание | Docker Compose, CPU-only, лимит 2GB RAM |

### pgvector-схема

```sql
-- Расширение pgvector
CREATE EXTENSION IF NOT EXISTS vector;

-- Таблица embeddings
CREATE TABLE embeddings (
  id SERIAL PRIMARY KEY,
  entity_type VARCHAR(50) NOT NULL,    -- subcontractor, review, event, checklist, ...
  entity_id INTEGER NOT NULL,          -- ID из основной таблицы
  chunk_index SMALLINT DEFAULT 0,      -- номер чанка (если запись разбита)
  content TEXT NOT NULL,               -- текст чанка (для BM25 и отображения)
  embedding VECTOR(768),               -- векторное представление чанка
  metadata JSONB DEFAULT '{}',         -- доп. поля (название, дата, связи)
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- HNSW-индекс для быстрого ANN-поиска
CREATE INDEX ON embeddings USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 200);
```

### Интеграция в основной backend

Основной backend (Node.js/Express порт 3001) проксирует запросы к AI-микросервису через **proxy-роут** `/api/ai`:

```
frontend → backend:3001/api/ai/ask  →  proxy  →  assistant:3002/api/ai/ask
```

Преимущества прокси:
- Фронтенд работает с единым origin (порт 3001), не нужен отдельный CORS
- Backend может добавить валидацию/логирование перед проксированием
- Простое конфигурирование: `AI_SERVICE_URL=http://localhost:3002`

При CRUD-операциях (создание/обновление/удаление любой сущности) backend отправляет **fire-and-forget** запрос на переиндексацию:
```
POST http://assistant:3002/api/ai/reindex/{entity}
```
Если микросервис недоступен — переиндексация откладывается до следующего полного перестроения индекса.

| Переменная | По умолчанию | Описание |
|------------|-------------|----------|
| `DATABASE_URL` | required | PostgreSQL (с pgvector) для таблицы `embeddings` |

| `LLM_API_URL` | `https://api.openai.com/v1` | OpenAI-совместимый API (заменяем на любой LLM-провайдер) |
| `LLM_MODEL` | `gpt-4o-mini` | Модель для генерации ответов |

| `PORT` | `3002` | Порт микросервиса |

### Безопасность

- Нет аутентификации (MVP) — `employeeId` передаётся в запросе как и в основном API
- Ассистент работает только на чтение — не модифицирует данные системы
- В промпт LLM передаются только обезличенные данные (ФИО сотрудников заменяются на идентификаторы)
- Доступ к `/api/ai/*` только изнутри Docker-сети — внешний доступ через прокси `/api/ai` основного backend
- Все ответы основаны только на данных системы — LLM не выдумывает факты (RAG constraint)

### Индексация данных

**Полная (при старте / по триггеру):**
1. Запрашивает все сущности из основного API (`GET /api/subcontractors`, `/api/reviews`, etc.)
2. Разбивает на чанки по смысловым единицам (одна запись → один чанк, длинные тексты → несколько)
3. Векторизует через `multilingual-e5-base` (768-dim)
4. Сохраняет в `embeddings` с метаданными (entity_type, entity_id, связи)

**Инкрементальная (на CRUD):**
- Backend отправляет fire-and-forget `POST /api/ai/reindex/{entity}` при каждом create/update/delete
- Микросервис переиндексирует только изменённую запись
