# Master Notepad — Описание системы

## Для чего эта система

**Master Notepad** — это единый источник правды о подрядчиках для строительной организации.

### Как было раньше (проблема)

Павел (руководитель) вёл блокнот, куда записывал все события по подрядчикам: нарушения протоколов совещаний, срывы сроков, хорошие и плохие поступки. Чтобы при конфликте было чем аргументировать.

Информация размазана по головам сотрудников, Excel-файлам, чатам в Telegram, бумажным блокнотам. Когда сотрудник увольняется — его опыт теряется безвозвратно. Когда нужно выбрать подрядчика на тендер — приходится обзванивать людей и по крупицам собирать отзывы. Протоколы совещаний ведутся в Excel делопроизводителем и живут отдельно от всего остального.

Типовые ошибки проектирования (например, «забыли указать места установки датчиков в проекте автоматизации») повторяются из проекта в проект, потому что негде их зафиксировать и превратить в чек-лист.

### Что делает система

Собирает **всё** в одном месте:
- Кто с кем работал и какие оценки поставил
- Хронологию событий по каждому подрядчику (сроки, нарушения, инциденты)
- Результаты проверок объектов (чек-листы)
- Протоколы совещаний с полным жизненным циклом и задачами
- Опросы коллектива для сбора коллективного мнения
- Рейтинг подрядчиков на основе накопленных данных
- AI-ассистент с RAG для быстрых ответов по данным системы

## Кто пользователи и их сценарии

### Павел (руководитель)

**Сценарий: Выбор подрядчика на тендер**
> «Нужен подрядчик на фундамент. Открываю систему, смотрю всех кто делает фундаменты. У каждого вижу: рейтинг, отзывы, хронологию событий (были ли срывы сроков?), протоколы совещаний. За 10 минут понимаю кому можно доверять, а кого не звать.»

**Сценарий: Разбор конфликта**
> «Подрядчик утверждает что мы не предупреждали о сроках. Открываю карточку подрядчика → журнал событий → 15 мая запись: "сорвал поставку арматуры на 3 дня". Показываю. Вопрос закрыт.»

**Сценарий: Накопление типовых ошибок**
> «В проекте автоматизации опять забыли датчики. Записываю это как событие. Система предлагает добавить в чек-лист проверки рабочей документации как типовую ошибку проектирования. В следующий раз чек-лист уже содержит этот пункт.»

### Инженер / Прораб (сотрудник)

**Сценарий: Приёмка работ**
> «Приехал на объект принимать фундамент. Открываю на телефоне чек-лист "Проверка фундамента", прохожу по пунктам, отмечаю выполненные. Вижу что два пункта не выполнены — тут же записываю событие о нарушении.»

**Сценарий: Передача опыта**
> «Работал с новым подрядчиком — оставил развёрнутый отзыв с оценкой. Коллеги теперь видят мой опыт, не нужно отдельно рассказывать.»

### Делопроизводитель

**Сценарий: Протокол совещания**
> «Прошло совещание с подрядчиком. Вместо Excel открываю систему, заполняю стандартную форму: повестка, участники, решения. Протокол автоматически привязывается к подрядчику. Через месяц любой сотрудник может его найти.»

## Функции системы

### 1. Картотека подрядчиков

База всех подрядчиков, с которыми работает или работала организация.

**Что хранится:**
- Название подрядчика (например, «СтройМонтаж»)
- Юридическое название (ООО «СтройМонтаж»)
- Контакты (телефон, email, адрес)
- Специализация (фундаменты, кровля, электрика, автоматизация...)
- Краткое описание

**Карточка подрядчика** собирает ВСЁ, что о нём известно:
- Отзывы сотрудников (оценка 1-10 + текст)
- **Журнал событий** — хронология всех инцидентов (см. раздел 2)
- Комментарии — заметки и доп. информация
- Протоколы совещаний с участием подрядчика
- Результаты опросов
- **Рейтинг** — рассчитывается автоматически (среднее отзывов ± вес событий)

### 2. Журнал событий

Хронология ВСЕХ значимых событий по каждому подрядчику.

**Чем отличается от отзыва:**
- **Отзыв** — это итоговая оценка («в целом хорошо, 8/10»)
- **Событие** — это конкретный факт с датой («15 мая 2025 — сорвал поставку арматуры на 3 дня»)

**Что содержит запись в журнале:**
- Дата события
- Тип события: ✅ положительное / ⚠️ нарушение / 📋 информационное
- Описание (свободный текст)
- Кто записал
- Связанный объект/проект (опционально) 🔜 будущее развитие — поле `project_id`, зарезервировано под FK → будущую таблицу `projects`

**Зачем это нужно:**
- **При конфликте** — железное доказательство: «15 мая вы сорвали поставку, вот запись»
- **При тендере** — посмотреть историю: сколько нарушений, какие именно
- **При анализе** — увидеть паттерны: подрядчик постоянно срывает сроки или это разовый случай
- **Накопление опыта** — типовые проблемы превращаются в пункты чек-листов

**Связь с чек-листами:**
Событие типа «типовая ошибка» может быть предложено к добавлению в организационный чек-лист. Например: событие «в проекте автоматизации не указаны места датчиков» → предложение добавить пункт «Проверить места установки датчиков» в чек-лист «Проверка рабочей документации».

### 3. Отзывы сотрудников

Итоговая оценка работы с подрядчиком от конкретного сотрудника.

**Что содержит:**
- Текст отзыва (конкретные факты, впечатления)
- Оценка от 1 до 10
- Кто оставил, когда

**Как используется для рейтинга:**
- Базовый рейтинг = среднее арифметическое всех отзывов
- Взвешенный рейтинг: учитывает события (+0.3 за positive, –0.5 за violation)

### 4. Чек-листы проверки объектов

Списки пунктов для проверки качества при приёмке работ.

**Два типа:**

| Тип | Кто создаёт | Кто использует | Пример |
|-----|------------|----------------|--------|
| **Организационные** | Любой сотрудник | Все (единый стандарт) | «Проверка фундамента», «Проверка рабочей документации» |
| **Личные** | Любой сотрудник | Только автор | Свои заметки и напоминалки |

**Операции:**
- Создать чек-лист с любым набором пунктов
- При проверке отмечать пункты ✅ выполнено / ❌ не выполнено
- Редактировать свои чек-листы

#### 4.1 Предложения по улучшению

Сотрудник «в поле» замечает что чек-лист неполный → предлагает улучшение → руководитель утверждает или отклоняет.

**Процесс:**
1. Сотрудник пишет: «Добавить проверку гидроизоляции в чек-лист фундамента»
2. Предложение получает статус «На рассмотрении»
3. В разделе «Предложения» руководитель видит все заявки
4. Одобряет (статус «Принято») или отклоняет («Отклонено»)

**Источники предложений:**
- Ручной ввод сотрудником
- Из журнала событий (событие типа «типовая ошибка» → предложение добавить в чек-лист)

### 5. Протоколы совещаний

Единая форма протокола, заменяющая Excel делопроизводителя. Протокол проходит через 5 этапов жизненного цикла от подготовки до рассылки.

#### Типы совещаний (уровни)

| Уровень | Описание | Пример |
|---------|----------|--------|
| Стратегическое | Совещание с руководством, не связано с конкретным проектом, касается работы подразделений в целом | Планирование квартальных KPI |
| Координационное | Внутреннее совещание команды проекта | Еженедельная планёрка проекта |
| Оперативное | Совещание с контрагентами (подрядчики, проектировщики, поставщики). Имеет подуровни по типам контрагентов | Совещание на площадке с подрядчиками |
| Проблемное | Совещание по конкретному вопросу | Разбор срыва сроков поставки |

#### Периодичность

- **Разовое** — первое совещание по теме. Разовое может быть преобразовано в периодическое.
- **Периодическое** — повторяющееся совещание (еженедельное, ежемесячное). Имеет сквозную нумерацию, историю протоколов, цепочку задач.

#### Структура протокола

Задачи внутри протокола могут быть сгруппированы одним из двух способов:

| Метод группировки | Описание |
|-------------------|----------|
| По темам | Группировка по разделам рабочей документации или направлениям (проектирование, СМР, экспертиза, ЗОС и т.д.) |
| По подрядчикам | Группировка по организациям-контрагентам |

#### Этап 1: Подготовка делопроизводителем

Делопроизводитель формирует:
- Повестку совещания (если проходит впервые)
- Дату, время, место проведения
- Список участников (сотрудники заказчика и представители контрагентов)
- Выполняет рассылку приглашений участникам

#### Этап 2: Подготовка контролирующими

Контролирующие (руководители строительства / директора по строительству):
- Создают структуру протокола (если первое совещание) или корректируют структуру (если периодическое)
- Формируют задачи и подзадачи к совещанию
- Назначают ответственных и сроки по каждой задаче
- Для периодических совещаний: ставят отметки о выполнении задач из предыдущего протокола
- Выполненные задачи переносятся в следующий протокол с отметкой «Выполнено», после чего уходят в архив

#### Этап 3: Ведение протокола

Делопроизводитель в ходе совещания:
- Фиксирует договорённости и принятые решения
- Делает заметки по ходу обсуждения
- По заметкам AI-ассистент предлагает задачи для включения в протокол
- Возможен «ленивый» вариант: создание протокола по стенограмме совещания (расшифровка аудиозаписи)
- Фиксирует явку участников, формирует список отсутствовавших

#### Этап 4: Утверждение протокола

- Ответственные по направлениям редактируют протокол (делопроизводитель может ошибаться в формулировках узких тем или пропускать вопросы)
- Каждый ответственный подтверждает свою часть протокола
- Переход к следующему этапу — только после утверждения ВСЕМИ ответственными
- Если по истечении установленного времени кто-то из ответственных не утвердил — протокол утверждается автоматически

#### Этап 5: Рассылка протокола

- Выполняется по клику после утверждения
- Адресаты: все участники совещания + отсутствовавшие
- Коммуникация с подрядчиками — только через email (доступа к системе у них нет)

#### Запрос задачи к следующему совещанию

Любой сотрудник может отправить запрос на добавление вопроса/задачи к повестке следующего совещания. Не привязан к конкретному этапу.

- Сотрудник создаёт запрос с описанием задачи
- Контролирующие рассматривают запрос
- При утверждении — задача включается в протокол
- При отказе — сотрудник получает уведомление с причиной

### 6. Задачи

Задачи — связующий элемент между протоколами совещаний. Позволяют отслеживать статус поручений и переносить задачи между протоколами разных уровней.

#### Нумерация и приоритет

- Каждая задача имеет уникальный сквозной номер
- Внутри протокола задачи нумеруются по порядку
- Порядок может меняться от совещания к совещанию: важные вопросы наверх, менее важные вниз

#### Переформулирование задачи

Задача может переформулироваться из протокола в протокол:

- **Уточнение** — формулировка уточняется по мере прояснения деталей
- **Цепочка подзадач** — большая задача разбивается на конкретные шаги
- Каждая подзадача может иметь собственного ответственного и контролирующего

#### Параллелизация задач между протоколами (ключевая возможность)

Задачу или подзадачу можно вручную перенести в другой протокол.

**Пример использования:**
> Оперативное совещание на площадке с подрядчиками. Проектировщики не присутствуют. На площадке поднят вопрос, касающийся проекта. Задачу переносят в протокол координационного совещания проектной команды. Решение, принятое на проектном совещании, подтягивается обратно в протокол оперативного совещания на площадке.

- Перенос выполняется **вручную** пользователем
- AI-ассистент **предлагает** кандидатов для переноса, но решение принимает человек
- При переносе сохраняется связь: видно из какого протокола задача пришла и куда ушла

#### Жизненный цикл задачи

```
Создана → В работе → Выполнена → В архив
```

- Выполненная задача появляется в следующем протоколе с отметкой «Выполнено»
- После отметки о выполнении уходит в архив

### 7. Комментарии

Дополнительная информация о подрядчике — «заметки на полях». Не отзыв и не событие.

**Примеры:**
- «Есть свой транспорт, может доставлять материалы»
- «Работают только по безналу»
- «Контактное лицо: Иван Иванович, +7 900...»
- «В среду перезвонить, уточнить по ЖК Восток»

### 8. Опросы сотрудников

Руководитель создаёт опрос о конкретном подрядчике → сотрудники отвечают → руководитель видит сводку.

**Стандартный шаблон (5 вопросов):**
1. Оцените качество работ подрядчика
2. Оцените соблюдение сроков
3. Оцените коммуникацию с подрядчиком
4. Общие впечатления о работе
5. Рекомендации и пожелания

**Важно:** Без конструктора — всегда 5 вопросов. Минимализм для быстрого сбора мнений.

### 9. Сотрудники

Список сотрудников организации. Нужен чтобы указывать автора отзыва, события, ответа на опрос.

**Данные:** ФИО, Email, Должность.

### 10. Участники и организации

#### Организации

В системе хранятся не только подрядчики (раздел 1), но и все организации-участники процесса:

| Поле | Описание |
|------|----------|
| Название | Наименование организации |
| ИНН | Идентификационный номер |
| Основной вид деятельности | Строительство, проектирование, поставки и т.д. |
| Связи с сотрудниками | Кто из сотрудников контрагента с нами работает |
| Связи с договорами | По каким договорам работает организация |

#### Сотрудники контрагентов

Представители подрядчиков, проектировщиков и других внешних организаций:

| Поле | Описание |
|------|----------|
| ФИО | Полное имя |
| Должность | Роль в организации |
| Специализация | У одной организации могут быть разные ответственные по направлениям |
| Контакты | Email, телефон |

### 11. Роли и доступ

| Роль | Права |
|------|-------|
| **Администратор** | Полный доступ ко всем функциям системы |
| **Делопроизводитель** | Создание и ведение протоколов, рассылка, фиксация явки |
| **Контролирующий (Ответственный)** | Формирование структуры протокола, создание задач, назначение ответственных, утверждение протокола, рассмотрение запросов задач |
| **Сотрудник Заказчика** | Только просмотр протоколов и задач (может создавать запросы задач) |
| **Подрядчики** | **Не имеют доступа к системе.** Коммуникация только через email (приглашения, рассылка протоколов) |

### 12. AI Ассистент

Интеллектуальный агент с RAG, который отвечает на вопросы в свободной форме по данным системы. Подробнее: [docs/product/assistant.md](assistant.md).

## Сценарии использования (Use Cases)

### UC1: Выбор подрядчика на тендер

1. Руководитель открывает раздел «Подрядчики»
2. Фильтрует по специализации («фундаменты»)
3. Видит список с рейтингами
4. Открывает карточку интересующего подрядчика
5. Смотрит: рейтинг, отзывы, журнал событий (нет ли нарушений)
6. Принимает решение кого звать на тендер
7. Нажимает «Тендерная справка» — получает сводку по подрядчику

### UC2: Разбор конфликта

1. Подрядчик утверждает что предупреждал о задержке
2. Руководитель открывает журнал событий подрядчика
3. Видит хронологию: записей о предупреждении нет
4. Показывает журнал → конфликт исчерпан

### UC3: Накопление типовых ошибок

1. Инженер обнаружил проблему: в проекте автоматизации не указаны места датчиков
2. Записывает событие с типом «⚠️ нарушение / типовая ошибка»
3. Система показывает кнопку «Предложить в чек-лист»
4. Инженер предлагает добавить пункт в чек-лист «Проверка рабочей документации»
5. Руководитель одобряет → пункт добавлен
6. При следующей проверке документации этот пункт уже есть

### UC4: Приёмка работ на объекте

1. Прораб на объекте, открывает телефон
2. Заходит в раздел «Чек-листы», выбирает «Проверка фундамента»
3. Проходит по пунктам, отмечает выполненные
4. Два пункта не выполнены → записывает событие о нарушении
5. Результаты проверки сразу видны руководителю

### UC5: Параллелизация задачи между протоколами

1. На оперативном совещании с подрядчиками на площадке поднят вопрос, требующий решения проектировщиков
2. Контролирующий вручную переносит задачу в протокол координационного совещания проектной команды
3. На проектном совещании задача обсуждается, принимается решение
4. Решение автоматически подтягивается обратно в протокол оперативного совещания на площадке
5. Все участники обоих протоколов видят связь и статус задачи
